Инструкция по запуску данного проекта.

1. Клонируем проект на локальную машину. 

```bash
    cd <'Путь в удобную для вас папку'>

    git clone https://github.com/Pr0st1k/Tasks_tracker.git

    cd <'Tasks_tracker'>
```

2. Установка зависимостей

```bash
    # npm
    npm install

    # pnpm
    pnpm install

    # yarn
    yarn install

    # bun
    bun install
```

3. Запуск json-server

```bash
    json-server --watch db.json --port 3001
```

4. Запуск сервера разработки

```bash
    # npm
    npm run dev

    # pnpm
    pnpm dev

    # yarn
    yarn dev

    # bun
    bun run dev
```

Краткое описание структуры проекта и особенностей реализации:

```bash
    Структура проекта: 
    За исключением стандартных папок от Nuxt 3 и Vue в проекте присутствуют следующие разделы: 
    •	layouts – папка со стандартными шаблонами, используемыми применяемые ко всем страницам (header/footer/sidebar)
    •	middleware – файлы защиты маршрутов
    •	pages – различные страницы
    •	server – файлы и папки JSON-сервера
    •	stores – файлы хранилищ Pinia

    Данный проект реализован c использованием следующих технологий: Nuxt 3, Vue.js, TailwindCSS, Pinia, bcryptjs, JWT и небольшого JSON-сервера

        При реализации проекта были внесены некоторые изменения в структуру предложенных эндпоинтов на сервере, а именно:

        1. Эндпоинты были разделены по тематическим папкам (auth/tasks/token)

        2. Добавлен эндпоинт для проверки валидности JWT-токена, который используется при авторизации пользователя, а также для проверки при попытках доступа к API

        3. Был заменен предложенный метод обновления задач. В задании был предложен способ обновления информации о задаче с использованием PUT-запросов, но использование данного типа запросов в данном контексте было принято нецелесообразным в связи с наличием полей, которые могут не передаваться в метод, а соответственно после получения ответа от сервера при отправке PUT-запроса, эти поля были бы удалены.

        4. Для улучшения пользовательского опыта, и ограничения действий до полной загрузки страниц был использован "лоадер-скрин".

    Основные задачи были выполнены. Сайт при регистрации и авторизации пользователя назначает ему accessToken, который используется для защиты страниц, и серверных API, путем отправки запросов с заголовком Authorization в котором передается токен. В дальнейшем при получении списка задач токен используется для получения userId, с целью вывести задачи определенного пользователя которому этот токен присвоен.

    В случае ошибок с авторизацией/действиями с задачами – перезапустить json server, командой «json-server --watch db.json --port 3001»
```
